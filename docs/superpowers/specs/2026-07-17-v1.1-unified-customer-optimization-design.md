# v1.1 統一 per-customer 最佳化評估端點設計

日期:2026-07-17
狀態:已核准方向,待實作
承接:靜態 SPA v1(旗艦「最佳化評估」頁目前客戶端拼裝三端點,數值不完全一致)。

## 背景與目標

v1 旗艦頁的三個面板來自三個引擎(售電/用電經濟=P2 優先序、逐案場=P3 MILP、時段=P4a),
數值不對齊。v1.1 新增**單一 per-customer 端點**,以 **P3 MILP 最佳化為唯一引擎**,
由**同一份分配**導出售電端、用電端、逐案場、時段別四塊,**必然一致**;並讓表單的
**RE 目標 / 綠電轉供價**即時覆寫。

方向決策(已拍板):
- 錨定 **P3 MILP 最佳化**(旗艦頁本就是最佳化評估,含 min% / 最少案場數)。
- 覆寫:`re_target_percent`(影響該客戶 RE 硬約束)、`transfer_price_per_kwh`(**單一**綠電
  轉供價,取代逐合約價算營收,對齊參考版單一費率)。
- 灰電用 flat `settings.grey_price_per_kwh`(月度經濟,同 P2 語意)。
- **時段別由月度最佳化分配推導**:各案場月度綠電分配 × 該案場時段發電占比 → 尖/半/離;
  時段 RE = 該時段綠電 ÷ 該時段用電。與經濟/逐案場**同源**,標註「依時段發電占比推估」。

## 後端

### 服務 `app/services/customer_optimization_service.py`
`compute_customer_optimization(db, customer_id, period, options) -> CustomerOptimizationResult`

`options`(dataclass 或 kwargs):`min_sites_per_customer`、`min_site_allocation_percent`、
`re_target_percent: float | None`、`transfer_price_per_kwh: float | None`。

步驟:
1. 載入 customer;不存在 → `NotFoundError`(→ 404)。
2. 建 optimizer 輸入(比照 `optimize_service`),但:**若 `re_target_percent` 覆寫非 None**,
   則該客戶 `CustomerDemand.green_target_type="re_percent"`、`re_target_percent=覆寫值`
   (其他客戶維持 DB 值;optimizer 為全域,只改此客戶的 RE 約束)。
3. `outcome = optimize_period(...)`(min_sites / min_pct 傳入)。
4. `focus = customer_id`。取該客戶 `allocated_mwh > 0` 的 allocations。
5. **由該份分配算經濟**(度數 = MWh×1000;`grey = settings.grey_price_per_kwh`;
   `feedin[farm] = farm.feed_in_price_per_kwh 或 settings.default_feed_in_price_per_kwh`;
   `price[c] = transfer_price 覆寫 或 contract.price_per_kwh 或 feedin`):
   - `green_kwh = Σ alloc×1000`
   - `procurement = Σ alloc×1000×feedin[farm(alloc)]`
   - `revenue = Σ alloc×1000×price(alloc)`
   - `gross_profit = revenue − procurement`;`gross_margin_percent = gp/revenue×100`(revenue=0→0)
   - `consumption = outcome.customer_summaries[focus].consumption_mwh`
   - `green_mwh = green_kwh/1000`;`grey_mwh = max(0, consumption − green_mwh)`
   - `re_percent = green_mwh/consumption×100`(consumption=0→0)
   - `avg_price = (revenue + grey_mwh×1000×grey) / (consumption×1000)`(consumption=0→0)
   - `added_cost = revenue − green_kwh×grey`
   - `used_default_feed_in_price` = 是否有 alloc 的 farm 無 feedin
6. **逐案場**:該客戶 allocations 依 `wind_farm_id` 加總 → `allocated_mwh`、
   `share_percent = alloc/總綠電×100`、`contract_number`(該場首筆)、`reason`、含
   `wind_farm_code`/`wind_farm_name`(由 WindFarm 查)。
7. **時段別(推導)**:`season = tou.season_of(period 月)`。
   - 各案場時段發電:`GenerationData`(period 內、`time_slot` 非空)依 farm×slot 加總 → `gen[farm][slot]`。
   - 各案場時段占比:`ratio[farm][slot] = gen[farm][slot] / Σ_slot gen[farm][*]`(該場無時段資料 → 平均三等分)。
   - 該客戶各案場月度綠電 `farm_alloc[farm]`(步驟 6)→ `green_slot[slot] = Σ_farm farm_alloc[farm]×ratio[farm][slot]`。
   - 各時段用電:`ConsumptionData`(period 內、`time_slot` 非空)該客戶依 slot 加總 → `con_slot[slot]`。
   - 每 slot:`re_percent = green_slot/con_slot×100`(con_slot=0→0);`grey_price_per_kwh = tou.grey_price(season, slot)`。
   - **不變式**:`Σ_slot green_slot == green_mwh`(捨入內);測試斷言。

### Schema `app/schemas/customer_optimization.py`
```
SellerSide{procurement_cost, sales_revenue, gross_profit, gross_margin_percent}
BuyerSide{total_consumption_mwh, green_mwh, grey_mwh, re_percent, avg_price_per_kwh, added_cost}
FarmAllocationOut{wind_farm_id, wind_farm_code, wind_farm_name, allocated_mwh, share_percent, contract_number, reason}
SlotRowOut{slot, grey_price_per_kwh, consumption_mwh, allocated_mwh, re_percent}
CustomerOptimizationResult{
  period, season, solver_status, customer_id, customer_code, company_name,
  re_target_percent, transfer_price_used(float|None),
  min_sites_per_customer, min_site_allocation_percent, used_default_feed_in_price,
  seller: SellerSide, buyer: BuyerSide,
  allocations: list[FarmAllocationOut], slot_breakdown: list[SlotRowOut]
}
```

### API(加於 `app/api/v1/analytics.py`)
```
GET /api/v1/analytics/customer-optimization
    ?customer_id=(必要,ge=1)
    &period=YYYY-MM(必要)
    &min_sites=(可選,ge=0,預設 settings.optimize_min_sites_per_customer)
    &min_site_allocation_percent=(可選,ge=0,le=100,預設 settings)
    &re_target_percent=(可選,ge=0,le=100)
    &transfer_price_per_kwh=(可選,ge=0)
→ 200 CustomerOptimizationResult;未知 customer → 404。
```

## 前端

- `web/api.js`:加
  `customerOptimization(customerId, period, minSites, minPct, reTarget, transferPrice)`。
- `web/app.js`:
  - 表單:**RE 目標 / 綠電轉供價改為可編輯 input**(預設:RE=該客戶 `re_target_percent`;
    轉供價=空→表示「依合約」,填值→覆寫)。選客戶時更新 RE 預設。
  - `runEvaluation`:改**單一** fetch `customerOptimization`;`renderResult` 直接消費 bundle。
  - **移除** v1 的三引擎拼裝與「數值來自不同引擎」免責註記;面板 aside 不再標不同引擎。
  - allocations 已含 farm 名稱 → 逐案場表不需另抓 wind-farms。
  - 時段表消費 `slot_breakdown`(已同源);slotnote 改為「依時段發電占比推估」。

## 測試策略

單元/整合 `tests/integration/test_customer_optimization.py`(用 `db` fixture,seed farms/
customers/contracts/gen(含時段)/consumption(含時段)):
- **一致性不變式**:`buyer.green_mwh == Σ allocations.allocated_mwh`(round);
  `re_percent == green_mwh/total_consumption×100`;`gross_profit == sales_revenue − procurement_cost`;
  `Σ slot_breakdown.allocated_mwh == buyer.green_mwh`。
- **覆寫**:傳 `transfer_price_per_kwh` → `sales_revenue == green_kwh×transfer_price`;
  傳 `re_target_percent` 提高 → 分配/RE 隨之變(或 shortfall 反映)。
- **邊界**:未知 customer → 404;該期間無資料 → 各 0、slot 空或全 0,不崩。
- API 整合 `tests/integration/test_customer_optimization_api.py`:seed → GET → 200 + 結構;
  帶覆寫參數 → 反映。
- 前端:`node --check` + 手動冒煙(表單覆寫 RE/轉供價 → 三面板一致)。

## 邊界與決策
- optimizer 全域,覆寫僅改 focus 客戶的 RE 目標;其他客戶不受影響。
- 時段推導為「時段發電占比」估計,非真實逐時段媒合(P4b 才是);UI 標註清楚。
- 灰電 flat(月度);時段表顯示 TOU 灰電價(資訊性,不影響月度均價)。
- v1 的 `/matching/optimize`、`/analytics/evaluation`、`/matching/slots` **不動**(其他用途仍在)。
